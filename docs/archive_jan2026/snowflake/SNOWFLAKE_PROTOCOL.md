# SNOWFLAKE Protocol — Scar Accumulation, Freeze, Partial Share

**Filed:** 2026-04-21
**Status:** Research protocol draft. Not cryptographically reviewed. Not shipping. Do not rely on this for any actual security property.
**Purpose:** Formalise the user-stated semantics — *"the snowflake is the prime scars from my first repo... once a lattice is filled with prime scars from computations that have run through it, the lattice is frozen as an internal encryption key, and parts of that 'snowflake' can be shared with other trusted users to create security networks, but the whole core snowflake is never shown"* — so a cryptographer can assess it and the funder pitch can cite a specific document.

**Relation to prior docs:**
- `TIG_SECURITY_ARCHITECTURE.md` (Jan 29 2026) defined the **4-layer architecture** (Lattice / Breath / Gauge / Gate) without spelling out the scar-freeze-share mechanics.
- `snowflake_null_spec.md` (this archive) defined **H₀ for the phase distribution** so the lattice's statistical signature is testable.
- This protocol document glues the two — how an accumulating scar set becomes a frozen key, and how parts can be shared without exposing the whole.

---

## 1. The scar

A **scar** is the minimal append-only record of one fire event:

```
scar := (phase, s_star, operator, timestamp, hw_class)
```

| Field | Type | Source |
|---|---|---|
| `phase` | int ∈ {0,…,12} | `breath.phase` at fire time (`crystalos.py` lines 160–177) |
| `s_star` | float ∈ [τ, 1] where τ = 0.7 | combined coherence at fire time (`crystalos.py` lines 109–143) |
| `operator` | int ∈ {0,…,9} | dominant TIG operator at fire time (`ck_tig.py` canon) |
| `timestamp` | int (ns since UNIX epoch) | local monotonic clock |
| `hw_class` | string enum: `{4core_constraint, 32core_abundance, …}` | machine-readable hardware-class label; fixed once at session start |

Fields chosen for the minimal set that lets a verifier:
(a) check the scar conforms to the null-spec (phase ∈ 0..12, s_star ≥ τ),
(b) compute the χ² signature from the full scar set,
(c) attribute the scar to a hardware class without leaking the individual machine.

**Invariants:**
- Scars are append-only. No mutation. No deletion.
- Scars within one session share `hw_class`.
- The scar set at any moment is totally ordered by timestamp.

## 2. Accumulation phase

Precondition: CRYSTALOS runtime is running, breath-gate state is cycling, gauge is computing `s_star`.

Loop (once per fire, driven by `crystalos.py` lines 281–283):
```
if breath.gate_open and s_star >= tau:
    scar = make_scar(phase=breath.phase, s_star=s_star,
                     operator=current_operator, timestamp=now_ns(),
                     hw_class=session_hw_class)
    scar_lattice.append(scar)
    fire_count += 1
```

**Termination:** accumulation continues until a **pre-registered target** is reached — either `fire_count == N0` or `wall_time == T0`, whichever comes first. `N0` and `T0` are declared at session start and logged in `session_manifest.json`. This closes the stopping-rule weakness flagged in `snowflake_null_spec.md` §7.

**Durations by hardware class (empirical):**

| hw_class | 50 Hz fires per min (observed) | N₀ = 67,297 takes | Source |
|---|---|---|---|
| 4core_constraint (Lenovo ThinkPad, Linux) | ~0.3 fires/min (geometry-bottlenecked) | weeks | `TIG_SECURITY_ARCHITECTURE.md` §8.1 (400 events ≈ days) |
| 32core_abundance (Dell R16, Windows) | ~10.9 fires/min | ≈ 4.3 days | `VERIFICATION_2026_04_21.md` (67,297 over Apr 17–21) |

These are observed rates, not guarantees. A fresh lattice on new hardware should do a **calibration pass** (target: ≥ 1000 fires) to estimate its own rate before declaring `N0`.

## 3. Freeze

At the pre-registered termination point, the lattice is **frozen** — made read-only and committed as a cryptographic object.

```
master_key := SHA-256( canonical_serialize( sorted_scar_set ) )
```

where `canonical_serialize` is a deterministic byte encoding of the scar set (scars sorted by timestamp; each scar encoded as a fixed-width record; session_manifest prepended).

**Properties:**
- `master_key` is a 256-bit hash.
- `master_key` is never transmitted off-device.
- Any subsequent write to `scar_lattice` raises `ReadOnlyError`.
- The canonical serialization is deterministic: the same scar set produces the same `master_key` bit-for-bit.

**Session manifest** (prepended to serialization):
```json
{
  "session_id": "<uuid>",
  "hw_class": "32core_abundance",
  "started_at": "2026-04-17T13:44:09Z",
  "ended_at":   "2026-04-21T19:04:XXZ",
  "n0_preregistered": 67297,
  "t0_preregistered": null,
  "crystalos_commit": "<git-sha-of-crystalos.py-at-session-start>",
  "null_spec_commit": "<git-sha-of-snowflake_null_spec.md-at-session-start>"
}
```

The commits are pinned so a future verifier can check that the scar set was generated by a specific, published version of the runtime and the null-spec.

## 4. Derived partial-shares (the sharing mechanism)

The user's constraint: *parts can be shared, the core is never shown.*

We use a **Merkle tree** over the sorted scar set.

```
leaf_i  := SHA-256( scar_i.encoded )                        # one leaf per scar
node    := SHA-256( left_child || right_child )             # internal nodes
root    := master_key                                       # proved identical to §3's hash
```

The root of this tree **equals** `master_key` (by construction). That gives us:

- Any scar can be proved a member of the lattice by supplying a `(scar_i, merkle_path_i)` pair that hashes up to `master_key`.
- `master_key` itself is published as a **commitment**; the scars are not.

**Partial-share operation:**

```
partial_share(recipient_id, fraction_f) returns:
    - shared_set S: a random f-fraction of the lattice's scars
    - merkle_proofs P: authentication paths for every scar in S
    - session_manifest M
    - (optional) zk_proof Π: proves "S is consistent with a valid
      crystalos.py run against snowflake_null_spec.md" without
      revealing anything about the remaining (1−f)-fraction
```

The recipient can:
1. Verify every scar in `S` is a genuine member (via merkle path to the published `master_key`).
2. Recompute the χ² of `S` and compare against the null-spec expectation — if `S` is a uniform random sample of a full abundance-regime lattice, its χ² should be within bounds.
3. NOT reconstruct the remaining (1−f)-fraction, because the hash is one-way and S is a random subset.

`f` is bounded: `0 < f ≤ f_max` where `f_max < 1` is a protocol parameter (candidate: `f_max = 0.1` — i.e., at most 10 % of scars are ever shareable). Without `f_max`, collusion of enough recipients reconstructs the lattice.

**ZK proof Π (sketch, not yet implemented):**
- Statement: "There exists a scar set L such that root(L) = master_key AND |L| = N_committed AND every scar in L satisfies (phase∈0..12, s_star≥τ, hw_class = declared) AND the χ² of L against the uniform-over-13-phases null is ≤ some public bound"
- Witness: the full scar set L (known to the prover, never transmitted)
- Candidate system: **zk-STARK over an arithmetisation of the scar constraints**. Each scar check is simple; the bottleneck is the root hash which STARKs handle well.

The ZK proof is the **missing cryptographic piece** — until it exists, partial-share discloses merkle paths but cannot prove "the non-shared fraction obeys the null-spec." That is acceptable for research-protocol exposition but not for security deployment.

## 5. Trust-network composition

Two parties A and B each hold their own frozen `master_key_A`, `master_key_B`. Each has (previously) shared a partial with the other: A gave B a slice `S_AB`; B gave A a slice `S_BA`.

**Mutual verification:**
- A verifies `S_BA` against `master_key_B`: every scar has a valid merkle path, and the χ² signature of `S_BA` is consistent with B's declared hw_class.
- B does the symmetric check on `S_AB`.

**Network property:** If A and B both verify, they have established a one-bit trust fact: *"we hold fragments of the same class of lattice (e.g. both 32-core abundance), generated by the same pinned crystalos.py commit, and the slices we hold of each other's lattices look statistically like legitimate CRYSTALOS runs."*

This is **not** a secret-sharing scheme (no reconstruction). It is an **attestation scheme**: each party attests to their own lattice by publishing `master_key`, and partial-shares let others audit without decrypting.

**Revocation:** since `master_key` is the root of a frozen tree, there is no revocation of a leaf. A party wanting to remove itself from the network publishes a **successor master_key** committed to a newly-frozen lattice, and the old one is marked superseded. The old network relationships remain valid as historical attestations.

## 6. Statistical signature — public, non-invertible

The χ² statistic of the full lattice against the 13-phase uniform null is a **hash-like summary**: one real number that captures the lattice's phase-distribution shape.

| Hardware class | Expected χ² (df=12) | Regime |
|---|---|---|
| 4core_constraint | > 21.0 (Lenovo: 22.03) | non-uniform — identity emerges from constraint |
| 32core_abundance | < 5.0 (Dell R16: 0.0353) | uniform — scheduler has no bottleneck |

A third party receiving a partial-share `S` can **verify hw_class claims** by computing χ²(S) and checking it falls in the expected band for the claimed class. This gives hw-class attestation *without* revealing scar contents.

Caveat: χ² bands must be empirically wide enough that a legitimate lattice falls inside. The bands for each hw_class are set by calibration sessions, not theory.

## 7. Threat model (informal — R4 deliverable spec)

**Adversary capabilities assumed:**
- Can observe `master_key` publications, partial-shares, and session manifests.
- Can compute SHA-256 and verify Merkle paths.
- Has access to the published `crystalos.py` source and `snowflake_null_spec.md`.
- Has access to a computer of their own and can run CRYSTALOS to generate their own lattice.
- Does NOT have access to the victim's machine or its `master_key`'s underlying scar set.

**Adversarial goals and current status:**

| Goal | Protocol protection | Status |
|---|---|---|
| Recover the full scar set from published `master_key` | SHA-256 preimage resistance | Standard crypto assumption — holds if SHA-256 holds. |
| Recover the non-shared (1−f) fraction from a partial-share | SHA-256 preimage + f ≤ f_max bound | Holds if partial shares are random subsets and f_max < 1. |
| Forge a `master_key` that passes the χ² check without running CRYSTALOS | ZK proof Π | **NOT YET** — Π is sketched only; an adversary could today publish a hash of a *synthetic* scar set that passes χ². |
| Forge a legitimate-looking partial-share | Merkle path falsifiability | Holds — forged shares would require SHA-256 collision. |
| Discover the hw_class of a lattice from `master_key` alone | Session manifest is pinned but a forger could lie about hw_class | Only partial shares let a verifier check hw_class via χ². |
| Stop-timing attack (learn something by observing *when* accumulation stops) | Pre-registered N₀ / T₀ | Holds provided N₀ / T₀ is declared and pinned before accumulation starts. |

The "Forge a `master_key`..." row is the **critical unresolved crypto work**. Without Π, the protocol's security property is "*if you trust the publisher ran CRYSTALOS honestly, the lattice behaves as claimed*." That is an honour-system property, not a cryptographic one.

## 8. What this protocol does NOT claim

- Does **not** claim post-quantum security (SHA-256 is the floor; a post-quantum merkle tree would substitute a hash-based signature scheme at the root).
- Does **not** claim that the χ² signature is unique to a lattice. Two different lattices can have the same χ². χ² is a *necessary* witness, not *sufficient identification*.
- Does **not** claim that `f_max` is safe at any specific value. Analysis of how much of the lattice can be revealed before the null signature is reconstructable from shares is a **research question**, not answered here.
- Does **not** claim that CRYSTALOS's hardware coupling is unforgeable. A sufficiently clever attacker could in principle emulate the 4-core hardware signature on more powerful hardware. Hardware-attestation primitives (TPM, SGX) are out of scope for this protocol but would layer cleanly on top.
- Does **not** claim that the Merkle-path partial-share prevents **cumulative** leakage if f_max is violated across many recipients.
- Does **not** claim that `master_key`'s publication reveals nothing about the lattice. It reveals the *hash* of the lattice, which is 256 bits of information.

## 10. What makes this worth doing

Conventional behavioural-biometric security ties an identity to a *pattern of use* — a typing rhythm, a gait, a click latency. SNOWFLAKE's ambition is one level deeper: the identity is tied to the **hardware's coherence signature under a cycle-driven gate**. Two things follow if this holds up to cryptographic review:

1. The authenticator is cheap — CRYSTALOS is 431 LOC of Python using only `psutil` and `nvidia-smi`.
2. The authenticator is **legible** — `master_key` is a hash, the partial-shares are merkle paths, the signature is a χ² number. Nothing is proprietary. A cryptographer can reason about it with standard tools.

These are the two properties that distinguish a research-grade security protocol from a product-marketing exercise. The protocol passes (1) today; it requires the Π work in §9 to pass (2) cleanly.
